# Symfony курс — с нуля до руководящей позиции

Интерактивный сайт для изучения Symfony: теория по официальной документации, задания с
выполнением PHP-кода в реальном времени в песочнице (в том числе внутри настоящего
Symfony-приложения), тесты на теорию и один сквозной проект — **BookShelf** (каталог книг),
который растёт вместе с курсом от простого JSON API до полноценного приложения.

## Структура репозитория

```
/web       Next.js 16 (App Router) — сам сайт курса
/sandbox   Node/Express-сервис, который выполняет PHP-код учеников в disposable Docker-контейнерах
/docker
  php-plain/     Dockerfile образа для голого PHP (без фреймворка)
  symfony-app/   Dockerfile образа с реальным Symfony-скелетом внутри
```

## Требования

- Node.js 20.9+
- Docker (запущенный Docker Desktop/daemon — песочница исполняет код внутри контейнеров)

## Первый запуск

```bash
npm install                     # ставит зависимости во всех workspace-пакетах
npm run sandbox:build-images    # собирает оба Docker-образа песочницы

npm run dev:sandbox             # поднимает песочницу на http://localhost:4000
npm run dev:web                 # в отдельном терминале — сайт на http://localhost:3000
```

Открой http://localhost:3000 (или порт, который Next.js выведет в консоли, если 3000 занят).

Если нужно указать сайту нестандартный адрес песочницы — скопируй `web/.env.local.example` в
`web/.env.local` и поменяй `SANDBOX_URL`.

## Как это устроено

- **Контент** живёт в `web/content/`: `manifest.ts` задаёт порядок блоков и уроков,
  `blocks/<block>/<lesson>.mdx` — сама теория (MDX, можно вставлять `<CodeExercise />` и
  `<Quiz />` прямо в текст), `exercises/` и `quizzes/` — определения заданий и тестов.
- **Песочница работает в двух режимах** (поле `mode` в определении упражнения,
  `web/content/types.ts`):
  - `plain-php` — код ученика выполняется как есть в одноразовом PHP-контейнере
    (`docker/php-plain`), без сети, с лимитами по памяти/CPU/PID и таймаутом 5с; сверяется
    stdout (`checks: Check[]`).
  - `symfony-app` — код ученика (один файл — контроллер, `.twig`-шаблон или Entity/Repository,
    путь задаётся `targetPath`) копируется в реальный Symfony-скелет внутри одноразового
    контейнера (`docker/symfony-app`). Если у упражнения заданы `setupCommands` (например,
    `doctrine:schema:create` + сидинг тестовых данных) — они выполняются до старта сервера.
    Дальше поднимается встроенный PHP-сервер, и по нему выполняются HTTP-запросы из `requests[]`
    — тоже изнутри контейнера, наружу сеть закрыта (`--network none`, loopback и файловая БД
    SQLite работают). Все запросы одного упражнения используют общий cookie jar внутри
    контейнера — сессия (flash-сообщения, в будущем — логин) переживает границу между запросами,
    как в настоящем браузере. У каждого запроса можно задать `contentType` (по умолчанию
    `application/json`, для form-urlencoded отправки — переопределить). Проверки —
    `checks: HttpCheck[]` по коду ответа и/или содержимому тела (`sandbox/src/symfony-app-runner.ts`).
    Образ `docker/symfony-app` также содержит постоянные «харнесс»-фикстуры
    (`docker/symfony-app/fixtures/`) — контроллеры, Entity, FormType и базовые шаблоны, которые
    не редактирует ученик (например, чтобы в упражнении на Twig ученик писал только `.twig`-файл,
    а в упражнении на формы — только FormType/Entity/контроллер, не всю инфраструктуру заново).
    Такие файлы не остаются невидимыми: упражнение может перечислить их в `contextFiles` —
    на странице появляется блок «Файлы проекта», где их можно раскрыть и прочитать (контент
    подтягивается с сервера через `GET /api/context-file?path=...` прямо из
    `docker/symfony-app/fixtures/`, один источник правды с тем, что реально исполняется).
  В обоих случаях: клиент отправляет только код на `POST /api/run` (Next.js), а `targetPath`,
  `requests` и `checks` берутся из определения упражнения на сервере — клиенту нельзя подделать
  результат.
- **Прогресс** хранится в `localStorage` браузера (без аккаунтов и бэкенда) — см. раздел «Дальше»
  ниже. Чтобы не потерять его при очистке данных браузера или перенести на другое устройство,
  в верхней панели есть кнопки скачать/загрузить прогресс в виде `.json`-файла.

## Как добавить новый урок

1. Написать `.mdx` файл в `web/content/blocks/<block>/<slug>.mdx`.
2. Если нужен интерактивный код — добавить определение в `web/content/exercises/` (выбрать
   `mode: "plain-php"` для голого PHP или `mode: "symfony-app"` для роутинга/контроллеров/API) и
   зарегистрировать в `web/content/exercises/index.ts`. Если упражнение опирается на уже
   существующие в песочнице файлы (Entity, FormType, харнесс-контроллер и т.п.) — перечислить их
   в `contextFiles`, чтобы ученик мог их открыть и прочитать прямо на странице.
3. Если нужен тест — добавить в `web/content/quizzes/` и зарегистрировать в
   `web/content/quizzes/index.ts`.
4. Добавить запись урока в `web/content/manifest.ts` (порядок в массиве = порядок в сайдбаре).

## Капстоун-проект и roadmap

Все мини-проекты курса — это один и тот же сервис BookShelf, который от блока к блоку обрастает
новыми возможностями (см. секцию «Капстоун-проект» на лендинге сайта — там расписан план по
каждому оставшемуся блоку). Сейчас реализованы:

- Блок 0 «Введение в Symfony» — установка, структура проекта (plain-PHP песочница).
- Блок 1 «Роутинг и контроллеры» — BookShelf, часть 1: JSON API каталога книг
  (symfony-app песочница).
- Блок 2 «Twig и шаблоны» — BookShelf, часть 2: HTML-витрина каталога (symfony-app песочница,
  харнесс-контроллеры из `docker/symfony-app/fixtures/`).
- Блок 3 «Doctrine ORM» — BookShelf, часть 3: та же JSON API, но на настоящей SQLite-базе
  вместо PHP-массива (symfony-app песочница, `setupCommands` для схемы БД и сидинга).
- Блок 4 «Формы и валидация» — BookShelf, часть 4: HTML-форма добавления книги, ограничения
  на Entity, flash-сообщения (symfony-app песочница, общий cookie jar на сессию).

Следующие блоки (security, DI вглубь, тестирование, console-команды, Messenger, кэширование,
деплой и финальный блок «Senior: подготовка к собеседованию») будут добавляться постепенно,
переиспользуя уже готовую инфраструктуру обоих режимов песочницы.

## Известные особенности окружения

- Если порт 3000 занят другим процессом, `next dev` автоматически возьмёт следующий свободный
  (обычно 3001) — смотри вывод команды.
- Упражнения в режиме `symfony-app` выполняются заметно дольше, чем `plain-php` (реальный
  boot Symfony-приложения плюс несколько HTTP-запросов) — обычно 1-3 секунды, для упражнений
  с `setupCommands` (Doctrine) — чуть больше. Общий таймаут на сервисе песочницы — 20 секунд на
  всю последовательность (настройка + запросы) одного упражнения.
