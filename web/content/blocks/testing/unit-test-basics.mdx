Весь курс до сих пор ты проверял свой код одним способом: запускал упражнение и смотрел на
результат глазами (или через автоматические HTTP-проверки этого сайта). В реальной разработке
так же важно уметь проверять код **без** сайта, браузера и ручного клика — автоматическими
тестами, которые может запустить кто угодно в любой момент. Инструмент для этого в PHP-мире —
**PHPUnit**.

## Юнит-тест: проверка одного класса в изоляции

"Юнит" (unit) — это, как правило, один класс или даже один метод. Юнит-тест не поднимает
Symfony-приложение целиком, не стучится в базу данных — он создаёт нужные объекты вручную и
проверяет, что метод возвращает то, что должен:

```php
namespace App\Tests\Service;

use App\Entity\Book;
use App\Service\BookFormatterService;
use PHPUnit\Framework\TestCase;

class BookFormatterServiceTest extends TestCase
{
    public function testFormatsTitleAndYear(): void
    {
        $book = new Book();
        $book->setTitle('Dune')->setYear(1965);

        $formatter = new BookFormatterService();

        $this->assertSame('Dune (1965)', $formatter->format($book));
    }
}
```

Разберём по частям:

- **`extends TestCase`** — базовый класс PHPUnit. Всё, что нужно для написания теста, приходит
  именно отсюда (все `assert*` методы — его методы).
- **Имя метода начинается с `test`** — так PHPUnit находит, какие методы класса запускать как
  тесты (остальные методы — обычные вспомогательные, если понадобятся).
- **`assertSame(ожидаемое, фактическое)`** — самая частая проверка: значения должны быть строго
  равны (`===`). Есть и другие: `assertTrue`, `assertNull`, `assertCount`, `assertInstanceOf` —
  весь список смотри в документации PHPUnit, когда понадобится что-то более специфичное.

## Три шага любого теста: Arrange, Act, Assert

Обрати внимание на структуру метода — она типична для подавляющего большинства тестов:

1. **Arrange** — подготовь данные (`$book = new Book(); ...`).
2. **Act** — вызови то, что тестируешь (`$formatter->format($book)`).
3. **Assert** — сравни результат с ожиданием (`assertSame(...)`).

Тест, в котором эти три шага перемешаны или не разделимы, обычно сложно читать — стоит держать их
явно друг за другом.

## Почему именно этот сервис — хороший пример для первого теста

`BookFormatterService` не обращается ни к базе данных, ни к HTTP-запросу, ни к какому-либо
сервису Symfony — у него нет зависимостей вообще (см. Блок 6). Именно такие классы проще всего
тестировать и именно с них стоит начинать: тест получается быстрым (доли миллисекунды, без
загрузки контейнера) и не требует никакой подготовки окружения.
