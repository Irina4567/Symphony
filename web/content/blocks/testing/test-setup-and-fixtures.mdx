Если в файле несколько тестов, и каждый начинается с одних и тех же трёх строк подготовки —
это сигнал вынести их в отдельное место. А если один тест незаметно зависит от данных, которые
оставил другой тест, — это уже не сигнал, а самая настоящая проблема: рано или поздно порядок
выполнения тестов поменяется, и всё сломается без единой строки изменений в самом коде.

## setUp(): подготовка перед каждым тестом

```php
class TestableBookApiControllerTest extends WebTestCase
{
    private KernelBrowser $client;

    protected function setUp(): void
    {
        $this->client = static::createClient();
    }

    public function testCreatedBookAppearsInList(): void
    {
        $this->client->request('POST', '/api/testable-books', ...);
        // ...
    }

    public function testCreateAndShowReturnSameTitle(): void
    {
        $this->client->request('POST', '/api/testable-books', ...);
        // ...
    }
}
```

`setUp()` — специальный метод, который PHPUnit вызывает автоматически **перед каждым** тестовым
методом класса (не один раз на весь класс!). Каждый тест получает свежий `$this->client`,
как будто он первый и единственный. Есть и парный метод `tearDown()` — выполняется **после**
каждого теста, для явной очистки (закрыть соединение, удалить временный файл) — в этом курсе он
почти не нужен, потому что каждый контейнер песочницы и так одноразовый.

## Тесты не должны зависеть друг от друга

Важная деталь примера выше: оба теста создают **свою собственную** книгу с уникальным названием
(`'Neuromancer'`, `'Snow Crash'`), а не полагаются на то, что уже лежит в базе. Почему это важно:

- PHPUnit **не гарантирует** порядок выполнения тестов (хотя по умолчанию часто идёт в порядке
  объявления, полагаться на это — ошибка).
- Если тест B молча рассчитывает на то, что тест A уже что-то создал, при запуске одного теста B
  в отдельности (например, при отладке через `--filter`) он сломается непонятным образом.
- Тесты, зависящие от порядка, невозможно безопасно запускать параллельно — а в реальных проектах
  параллельный прогон тестов — обычная практика ради скорости CI.

## Фикстуры — где проходит граница

В этом курсе тестовые данные создаются прямо внутри теста (или через `setupCommands` для общей
схемы БД, но не общих данных, специфичных для конкретных тестов). В больших проектах для этого
часто используют отдельные библиотеки фикстур (`doctrine/data-fixtures`), которые умеют
воспроизводимо наполнять базу заранее описанным набором данных. Принцип везде один: тестовые
данные должны быть **явными и предсказуемыми**, а не "тем, что случайно осталось от прошлого
теста".
