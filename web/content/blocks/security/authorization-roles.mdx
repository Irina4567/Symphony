Аутентификация отвечает на вопрос «кто ты?». Следующий вопрос — «что тебе разрешено?» — это уже
**авторизация**. В Symfony у неё есть два основных механизма: декларативный (в конфиге) и
императивный (в коде). Разберём оба на примере одного и того же приложения.

## Роли: что это такое

Роль — это просто строка вида `ROLE_USER`, `ROLE_ADMIN`. Метод `getRoles()` в Entity `User`
(см. прошлый урок) возвращает массив таких строк для конкретного пользователя, и всегда добавляет
`ROLE_USER` — эта роль есть у любого залогиненного пользователя автоматически. Никакой встроенной
иерархии ролей в этом курсе мы не используем (в реальных проектах она настраивается отдельно
через `role_hierarchy`), поэтому `ROLE_ADMIN` не включает `ROLE_USER` сама по себе — их надо
задавать явно, как в `SeedUserCommand`.

## Способ 1: access_control — декларативная защита по URL

```yaml
security:
    access_control:
        - { path: ^/secure/admin-report, roles: ROLE_ADMIN }
```

Каждое правило — это регулярное выражение по пути и список ролей, которые для него требуются.
Правила проверяются сверху вниз, срабатывает первое совпадение. Плюс подхода — вся политика
доступа видна в одном файле, без необходимости заглядывать в каждый контроллер. Минус — не видно
прямо в коде контроллера, что маршрут вообще защищён, если не открыть `security.yaml` рядом.

## Способ 2: #[IsGranted] — защита прямо в контроллере

```php
#[Route('/secure/my-profile', methods: ['GET'])]
#[IsGranted('ROLE_USER')]
public function __invoke(): JsonResponse
{
    return new JsonResponse(['email' => $this->getUser()?->getUserIdentifier()]);
}
```

Атрибут над методом (или над всем классом — тогда действует на все его маршруты). Плюс — защита
видна сразу рядом с самим кодом эндпоинта. Оба способа опираются на одну и ту же систему ролей и
работают полностью равнозначно — выбор между ними обычно вопрос стиля команды, а не технических
ограничений.

Тот же самый эффект можно получить и без атрибута, вызовом прямо внутри метода:

```php
public function __invoke(): JsonResponse
{
    $this->denyAccessUnlessGranted('ROLE_USER');
    // ...
}
```

Это полезно, когда решение о доступе зависит от чего-то, что известно только в теле метода —
например, от конкретного объекта (об этом — в следующем уроке про Voters).

## 401 vs 403 — Symfony расставляет их автоматически

- **401 Unauthorized** — "мы не знаем, кто ты": анонимный запрос к защищённому маршруту.
- **403 Forbidden** — "мы знаем, кто ты, но тебе сюда нельзя": пользователь аутентифицирован, но
  нужной роли нет.

Оба кода возникают из одного и того же исключения `AccessDeniedException` — Symfony сама решает,
какой код вернуть, в зависимости от того, был ли запрос вообще аутентифицирован. Разработчику не
нужно выбирать код руками ни при `access_control`, ни при `#[IsGranted]`, ни при
`denyAccessUnlessGranted()` — поведение одинаковое во всех трёх случаях.
