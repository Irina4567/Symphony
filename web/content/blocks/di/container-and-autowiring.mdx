Всю дорогу через прошлые пять блоков ты уже пользовался Dependency Injection, даже не называя
это так: `EntityManagerInterface $em`, `UserPasswordHasherInterface $hasher` — эти объекты
появлялись в конструкторах и методах контроллеров сами, без единого `new`. Настало время
разобраться, как это устроено — и написать первый **собственный** сервис.

## Что такое контейнер служб (service container)

Контейнер — это объект, который знает, как создать любой сервис приложения: EntityManager,
твой собственный класс, что угодно. Вместо того чтобы создавать зависимости вручную
(`new BookFormatterService()`) внутри метода, ты **запрашиваешь** их — через параметр
конструктора или, для контроллеров, прямо через параметр метода-экшна. Контейнер сам разбирается,
что и в каком порядке создать.

## Автовайринг: почему это просто работает

```php
namespace App\Service;

use App\Entity\Book;

class BookFormatterService
{
    public function format(Book $book): string
    {
        return sprintf('%s (%d)', $book->getTitle(), $book->getYear());
    }
}
```

Этот класс — уже полноценный сервис, хотя нигде не зарегистрирован вручную. По умолчанию
`config/services.yaml` любого нового проекта Symfony содержит:

```yaml
services:
    _defaults:
        autowire: true
        autoconfigure: true

    App\:
        resource: '../src/'
```

- **`resource: '../src/'`** — все классы в `src/` автоматически становятся сервисами контейнера.
- **`autowire: true`** — если параметр конструктора/метода типизирован конкретным классом
  (или интерфейсом — см. следующий урок), контейнер сам найдёт подходящий сервис по типу и
  передаст его. Тебе не нужно ничего прописывать вручную для каждого нового класса.
- **`autoconfigure: true`** — сервисы автоматически получают нужные теги по тому, какие
  интерфейсы они реализуют (именно так, например, в Блоке 5 автоматически подхватывался твой
  `Voter`, без единой строчки конфига).

## Внедрение в контроллер — тот же механизм

```php
#[Route('/books/formatted/{id}', methods: ['GET'])]
public function __invoke(int $id, EntityManagerInterface $em, BookFormatterService $formatter): JsonResponse
{
    $book = $em->getRepository(Book::class)->find($id);
    // ...
    return new JsonResponse(['formatted' => $formatter->format($book)]);
}
```

Метод-экшн контроллера — тоже часть механизма автовайринга: любой параметр, который не пришёл
из самого запроса (как `$id` из `{id}` в пути), Symfony пытается разрешить через контейнер по
типу. Именно поэтому `EntityManagerInterface $em` во всех прошлых блоках просто появлялся сам —
это не магия конкретно EntityManager, а общий механизм, который работает для абсолютно любого
класса в `src/`.

## Почему не просто `new BookFormatterService()`

Кажется, что для такого простого класса можно было бы обойтись без DI вообще. Но с ростом
проекта у сервисов появляются собственные зависимости (репозитории, другие сервисы, конфигурация
через переменные окружения) — создавать всё это вручную в каждом контроллере, где оно нужно,
означает дублировать логику сборки объекта по всему проекту. Контейнер один раз знает, как
собрать `BookFormatterService`, и переиспользует эту логику везде, где он нужен — включая тесты,
где реальный сервис легко подменить на облегчённую версию.
