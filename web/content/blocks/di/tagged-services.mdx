Явное связывание из прошлого урока отлично решает задачу "мне нужна именно эта одна
реализация". Но что если нужна **не одна конкретная**, а **сразу все** реализации интерфейса —
например, чтобы показать пользователю список всех доступных стратегий рекомендации? Перечислять
их вручную одну за другой неудобно и легко забыть добавить новую. Для этого в Symfony есть
**теги**.

## Тег — это просто метка на сервисе

Тег — строка вроде `'app.recommendation_strategy'`, которую можно повесить на любой сервис. Затем
можно попросить контейнер: "дай мне ВСЕ сервисы с этим тегом" — одним параметром, без
перечисления классов.

Вешать тег на каждую реализацию вручную было бы так же неудобно, как перечислять их в коде.
Вместо этого тег вешают один раз — на сам **интерфейс**:

```php
namespace App\Service;

use App\Entity\Book;
use Symfony\Component\DependencyInjection\Attribute\AutoconfigureTag;

#[AutoconfigureTag('app.recommendation_strategy')]
interface RecommendationStrategyInterface
{
    public function name(): string;

    /** @param Book[] $books */
    public function recommend(array $books): ?Book;
}
```

Благодаря `autoconfigure: true` (см. Урок 1) **любой** класс, реализующий этот интерфейс,
автоматически получает тег `app.recommendation_strategy` — включая классы, которые появятся в
будущем. Это тот же самый механизм, который в Блоке 5 без единой строчки конфига подхватывал
твой `Voter` — Voter тоже помечен таким тегом внутри самого Symfony.

## Получение всех сервисов по тегу

```php
public function __invoke(#[AutowireIterator('app.recommendation_strategy')] iterable $strategies): JsonResponse
{
    $names = [];
    foreach ($strategies as $strategy) {
        $names[] = $strategy->name();
    }

    return new JsonResponse(['strategies' => $names]);
}
```

`#[AutowireIterator('тег')]` — ещё один именованный вариант `#[Autowire]` (как и `service:` из
прошлого урока), только вместо одного сервиса он внедряет **итератор** по всем сервисам с этим
тегом. Строка тега в `#[AutowireIterator(...)]` должна совпадать со строкой в
`#[AutoconfigureTag(...)]` — это просто одна и та же метка с двух сторон.

## Почему это лучше, чем просто внедрить массив классов

Если в проект добавят третью стратегию (скажем, "по любимому автору пользователя"), тегированный
список подхватит её автоматически — код, который перебирает `$strategies`, менять не придётся.
Явное перечисление классов пришлось бы редактировать при каждой новой стратегии; теги переносят
эту заботу с "каждого места использования" на "автоконфигурацию одного интерфейса".
