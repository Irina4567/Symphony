Все команды этого блока до сих пор возвращали `Command::SUCCESS` без вариантов — но реальные
команды не всегда справляются: файла нет, книга не найдена, соединение с внешним сервисом упало.
Последний урок блока — про то, как честно сообщать об этом и человеку в терминале, и скрипту,
который вызвал команду автоматически.

## SymfonyStyle: оформленный вывод поверх OutputInterface

```php
namespace App\Command;

use App\Entity\Book;
use Doctrine\ORM\EntityManagerInterface;
use Symfony\Component\Console\Attribute\AsCommand;
use Symfony\Component\Console\Command\Command;
use Symfony\Component\Console\Input\InputArgument;
use Symfony\Component\Console\Input\InputInterface;
use Symfony\Component\Console\Output\OutputInterface;
use Symfony\Component\Console\Style\SymfonyStyle;

#[AsCommand(name: 'app:book-lookup')]
class BookLookupCommand extends Command
{
    public function __construct(private readonly EntityManagerInterface $em)
    {
        parent::__construct();
    }

    protected function configure(): void
    {
        $this->addArgument('title', InputArgument::REQUIRED, 'Название книги для поиска');
    }

    protected function execute(InputInterface $input, OutputInterface $output): int
    {
        $io = new SymfonyStyle($input, $output);
        $title = $input->getArgument('title');

        $book = $this->em->getRepository(Book::class)->findOneBy(['title' => $title]);

        if (!$book) {
            $io->error("Книга не найдена: {$title}");
            return Command::FAILURE;
        }

        $io->success("Найдена: {$book->getTitle()} ({$book->getYear()})");
        return Command::SUCCESS;
    }
}
```

`new SymfonyStyle($input, $output)` не заменяет `$output` — она оборачивает его и добавляет готовые
оформленные блоки: `$io->success(...)` (зелёный блок), `$io->error(...)` (красный), `$io->warning(...)`,
а также `$io->table(...)`, `$io->listing(...)` для структурированного вывода. Это тот же принцип,
что и `SymfonyStyle` в других местах фреймворка — не новая концепция, а удобная обёртка.

## Два разных получателя одного результата

У команды всегда два адресата:

- **Человек**, читающий терминал — для него текст `$io->success(...)`/`$io->error(...)`: понятно
  на естественном языке, что произошло.
- **Скрипт или CI-пайплайн**, вызвавший команду — для него единственный сигнал это **код
  завершения** (`$?` в bash сразу после вызова). Он не будет парсить текст на русском, чтобы
  понять, успех это или нет.

```bash
php bin/console app:book-lookup "1984"
echo $?   # 0 — Command::SUCCESS

php bin/console app:book-lookup "Nonexistent"
echo $?   # 1 — Command::FAILURE
```

## Почему нельзя просто вывести ошибку и вернуть SUCCESS

Если `execute()` при ненайденной книге выведет `$io->error(...)`, но вернёт `Command::SUCCESS`,
любая автоматизация поверх этой команды (`app:book-lookup "$title" || exit 1` в деплой-скрипте)
решит, что всё прошло штатно — код 0 для неё важнее текста. Ошибка "потеряется" ровно там, где она
важнее всего: в CI, где никто не читает вывод построчно. Код завершения и текст сообщения — два
независимых канала, и оба должны быть честными.
