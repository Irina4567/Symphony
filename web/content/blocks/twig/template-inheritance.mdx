Каждая страница сайта — это не отдельный, независимый HTML-документ. У них общая шапка, футер,
подключение стилей — и дублировать это в каждом шаблоне означало бы менять один и тот же код в
десятке мест при любой правке. Наследование шаблонов решает эту задачу так же, как наследование
классов решает дублирование кода в PHP.

## Базовый шаблон

```twig
{# templates/base.html.twig #}
<!DOCTYPE html>
<html>
<head>
    <title>{% block title %}BookShelf{% endblock %}</title>
</head>
<body>
    <main>{% block content %}{% endblock %}</main>
</body>
</html>
```

`{% block title %}...{% endblock %}` объявляет именованный блок со значением **по умолчанию**
(«BookShelf») — если дочерний шаблон его не переопределит, останется именно это значение.

## Дочерний шаблон

```twig
{# templates/books/list.html.twig #}
{% extends 'base.html.twig' %}

{% block title %}Каталог книг{% endblock %}

{% block content %}
    <ul>
        {% for book in books %}
            <li>{{ book.title }}</li>
        {% endfor %}
    </ul>
{% endblock %}
```

`{% extends %}` обязательно должен идти первой инструкцией в файле (до него допустимы только
комментарии). Дальше — только переопределение блоков; весь остальной HTML уже описан в родителе.

## parent() — дополнить, а не заменить

Если нужно не полностью заменить содержимое блока, а добавить что-то к уже существующему —
внутри переопределённого блока можно вызвать `{{ parent() }}`:

```twig
{% block javascripts %}
    {{ parent() }}
    <script src="/books.js"></script>
{% endblock %}
```

## Зачем это в реальном проекте

Наследование — это не просто «красивее», а конкретная защита от бага: без него любое изменение
общей разметки (например, добавить баннер cookie-согласия) означает правку в каждом .twig-файле
проекта. С наследованием — правку одного base.html.twig.
