Последний кусочек перед мини-проектом: как правильно ссылаться на другие страницы и статические
файлы — не хардкодя пути строками.

## path() — ссылка на маршрут по имени

Мы уже видели `generateUrl()` в контроллере (Блок 1) — `path()` делает то же самое, но внутри
Twig-шаблона:

```twig
<a href="{{ path('books_show', {id: book.id}) }}">{{ book.title }}</a>
```

Обрати внимание на синтаксис: `{id: book.id}` внутри `{{ }}` — это Twig-хэш (аналог
ассоциативного массива), а не блок `{% %}`. Если маршрут `books_show` объявлен как
`/books/{id}`, получится `/books/42`.

## Почему не через хардкод

```twig
{# Так делать не стоит: #}
<a href="/books/{{ book.id }}">{{ book.title }}</a>
```

Работать будет ровно до того момента, пока кто-то не поменяет путь маршрута в контроллере —
тогда все хардкоженные ссылки по всему проекту молча сломаются (по ним будет 404), а
`path('books_show', ...)` пересоберёт правильный URL автоматически, потому что берёт его из той
же таблицы маршрутов, что и сам роутинг.

## url() — абсолютный адрес

`path()` возвращает относительный путь (`/books/42`) — этого достаточно для ссылок внутри своего
же сайта. Когда нужен полный адрес с доменом и схемой (например, в письме или мета-теге
OpenGraph) — используется `url()`:

```twig
{{ url('books_show', {id: book.id}) }}
{# https://bookshelf.example.com/books/42 #}
```

## asset() — ссылки на статику

Для файлов из `public/` (CSS, JS, изображения) — своя функция:

```twig
<link rel="stylesheet" href="{{ asset('styles/app.css') }}">
```

Та же идея, что и с `path()`: не собирать путь руками, а дать фреймворку сделать это правильно
(в реальных проектах `asset()` умеет добавлять версию файла в URL для инвалидации кэша браузера).
