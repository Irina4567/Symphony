`PingEvent` из прошлого урока — учебная игрушка с одним публичным полем. Настоящие события несут
осмысленные данные о том, что произошло. Спроектируем такое событие для BookShelf:
"книга была создана".

## Что должно нести событие

`BookCreatedEvent` должно как минимум давать слушателям доступ к созданной книге:

```php
namespace App\Event;

use App\Entity\Book;

class BookCreatedEvent
{
    public function __construct(private readonly Book $book)
    {
    }

    public function getBook(): Book
    {
        return $this->book;
    }
}
```

`readonly` здесь не случаен: после публикации событие описывает уже случившийся факт — какая
книга была создана, — и меняться эта информация не должна. Слушатели читают её, не переписывают.

## А что, если слушателям нужно "ответить" событию?

Иногда полезно, чтобы слушатель мог сообщить о своей работе — не поменять сам факт (книга
создана — это неизменяемо), а добавить **дополнительную** информацию, которую потом прочитает
код, опубликовавший событие. Для этого у события может быть отдельное **мутируемое** состояние,
явно для этой цели:

```php
class BookCreatedEvent
{
    private bool $notified = false;
    private array $log = [];

    public function __construct(private readonly Book $book) {}

    public function getBook(): Book { return $this->book; }

    public function markNotified(): void { $this->notified = true; }
    public function isNotified(): bool { return $this->notified; }

    public function log(string $entry): void { $this->log[] = $entry; }
    public function getLog(): array { return $this->log; }
}
```

Так `$book` остаётся неизменным описанием факта, а `$notified`/`$log` — это канал связи "слушатель
→ код, опубликовавший событие", отдельный и явный. В реальных проектах такое нужно нечасто (обычно
слушатели просто делают своё дело — отправляют письмо, пишут в БД — и никому не отчитываются), но
в этом курсе это ещё и позволяет проверить, что слушатель действительно сработал, через обычный
HTTP-ответ, без необходимости лезть в базу или почтовый ящик за подтверждением.

## Событие без единого слушателя — это нормально

```php
$event = new BookCreatedEvent($book);
$dispatcher->dispatch($event);
// $event->isNotified() === false, $event->getLog() === []
// и это не ошибка — просто пока никто не подписан
```

`dispatch()` не требует, чтобы у события был хотя бы один слушатель — если подписчиков нет, вызов
просто ничего не делает и возвращает тот же объект без изменений. Раздельная разработка события и
его слушателей — совершенно нормальный порядок: сначала описываем факт, потом (в следующих уроках)
решаем, кто и как на него реагирует.
