Одно событие может слушать сколько угодно классов. Но что если порядок их срабатывания важен —
например, один слушатель должен провалидировать данные до того, как другой отправит уведомление?
За порядок отвечает **priority**.

## Приоритет: чем выше число, тем раньше срабатывает

```php
#[AsEventListener(event: BookCreatedEvent::class, priority: 10)]
public function onBookCreated(BookCreatedEvent $event): void
{
    $event->log('priority-first');
}
```

```php
// другой класс, приоритет не указан — по умолчанию 0
#[AsEventListener(event: BookCreatedEvent::class)]
public function onBookCreated(BookCreatedEvent $event): void
{
    $event->log('audit');
}
```

При публикации `BookCreatedEvent` сначала сработает слушатель с приоритетом `10`, и только потом —
с приоритетом `0` (по умолчанию). Результат: `$event->getLog()` вернёт
`['priority-first', 'audit']` — именно в этом порядке.

Приоритет может быть и отрицательным — это удобно для слушателей, которые должны срабатывать
последними среди всех (например, что-то вроде финального логирования "событие полностью
обработано").

## Зачем это нужно на практике

Типичный пример из реальных проектов: один слушатель на событие "форма отправлена" проверяет права
доступа и может остановить дальнейшую обработку (`stopPropagation()`, если событие унаследовано от
базового `Event`), а другой, с более низким приоритетом, — уже отправляет уведомление. Без явного
приоритета порядок срабатывания слушателей не гарантирован и может незаметно поменяться, если
кто-то в команде допишет ещё один слушатель на то же событие.

## Приоритет — это порядок, а не эксклюзивность

Важно не путать приоритет с "первый сработавший останавливает остальных" — по умолчанию **все**
подписанные слушатели срабатывают, просто в порядке приоритета. Чтобы буквально прервать цепочку
раньше времени, нужен отдельный явный вызов `stopPropagation()` на событии, унаследованном от
`Symfony\Contracts\EventDispatcher\Event` — но это довольно редкий случай, и для большинства задач
достаточно просто порядка.
