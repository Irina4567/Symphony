До сих пор контроллер сам решал всё, что должно произойти: сохранить книгу, сформировать ответ.
Но что если "что-то ещё" должно произойти после основного действия — например, залогировать факт
создания книги — а сам контроллер знать об этом не должен, чтобы не разрастаться и не зависеть
от деталей логирования? Для развязки таких вещей в Symfony есть **EventDispatcher** — реализация
паттерна publish/subscribe.

## Идея: публикация вместо прямого вызова

Вместо того чтобы контроллеру самому вызывать `$logger->log(...)`, `$mailer->send(...)` и всё, что
ещё может понадобиться, он просто **публикует событие** — "вот что произошло" — и не заботится о
том, кто и как на это отреагирует. Слушатели подписываются на события заранее, а диспетчер сам
находит и вызывает нужных.

## Событие — обычный PHP-объект

```php
namespace App\Event;

class PingEvent
{
    public function __construct(public string $message)
    {
    }
}
```

В отличие от многих других фреймворков, событию в Symfony **не нужно** наследоваться от особого
базового класса — подойдёт любой обычный объект. (Базовый класс `Event` существует и добавляет
метод `stopPropagation()` — остановить дальнейшую цепочку слушателей, но для большинства случаев
он не нужен.)

## Слушатель: реагирует на конкретный тип события

```php
namespace App\EventListener;

use App\Event\PingEvent;
use Symfony\Component\EventDispatcher\Attribute\AsEventListener;

class PingEventListener
{
    #[AsEventListener(event: PingEvent::class)]
    public function onPing(PingEvent $event): void
    {
        $event->message .= ' (heard!)';
    }
}
```

`#[AsEventListener(event: ...)]` — атрибут, ничем не отличающийся по духу от `#[AsCommand]` или
`#[Route]`: он превращает обычный метод в подписчика на конкретное событие. Регистрировать его
вручную в конфиге не нужно — `autoconfigure: true` (см. Блок 6) сам находит все такие методы.

## dispatch(): синхронный вызов всех подписчиков

```php
public function __invoke(EventDispatcherInterface $dispatcher): JsonResponse
{
    $event = new PingEvent('hello');
    $dispatcher->dispatch($event);

    return new JsonResponse(['message' => $event->message]);
    // 'hello (heard!)' — слушатель успел сработать до этой строки
}
```

Важная деталь: `dispatch()` вызывает всех слушателей **синхронно и по очереди**, прямо во время
самого вызова — это не "поставить в очередь на потом". Если слушатель не бросает исключение, к
моменту, когда `dispatch()` вернёт управление, все подписанные слушатели уже отработали. Именно
поэтому контроллер может сразу прочитать `$event->message` после `dispatch()` — событие могло
быть изменено любым из слушателей.
