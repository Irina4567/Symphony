Событие из прошлого урока живёт своей жизнью, но пока ничего не происходит после его публикации.
Пришло время написать слушателя, который на него реагирует.

## Слушатель — обычный класс с одним методом

```php
namespace App\EventListener;

use App\Event\BookCreatedEvent;
use Symfony\Component\EventDispatcher\Attribute\AsEventListener;

class BookCreatedNotifier
{
    #[AsEventListener(event: BookCreatedEvent::class)]
    public function onBookCreated(BookCreatedEvent $event): void
    {
        $event->markNotified();
        // в реальном проекте здесь обычно письмо, запись в БД или лог,
        // а не мутация самого события
    }
}
```

Ничего специального регистрировать не нужно: `autoconfigure: true` подхватывает метод, помеченный
`#[AsEventListener]`, точно так же, как в Блоке 5 подхватывался `Voter`, а в Блоке 6 —
любая реализация помеченного тегом интерфейса. Это один и тот же принцип автоконфигурации,
применённый к разным задачам.

## Имя метода не имеет значения

```php
#[AsEventListener(event: BookCreatedEvent::class)]
public function whateverNameYouWant(BookCreatedEvent $event): void { /* ... */ }
```

В отличие от некоторых других механизмов Symfony (например, геттеров/сеттеров Entity, которые
Doctrine ищет по имени), имя метода слушателя — на усмотрение разработчика. Важна только связка:
атрибут указывает, на какое событие подписан МЕТОД, а тип параметра должен совпадать с классом
события (или соответствовать тому, что указано в `event:`).

## В реальности слушатель обычно не мутирует событие

`markNotified()` в этом курсе — учебный приём, чтобы результат работы слушателя можно было
проверить прямо в HTTP-ответе. На практике типичный слушатель делает что-то "во внешний мир":
отправляет письмо через `MailerInterface`, пишет строку в лог через `LoggerInterface`, сохраняет
запись через `EntityManagerInterface` — и ничего не возвращает и не меняет в самом событии. Дизайн
конкретно `BookCreatedEvent` с его `notified`/`log` — способ сделать side effect наблюдаемым для
целей обучения, а не универсальный паттерн для копирования во все свои события.

## Что если слушателей несколько?

Ничего не мешает подписать на одно и то же событие сколько угодно классов — каждый получит свою
копию (точнее, ссылку на тот же объект события) и сможет прочитать или изменить его состояние.
