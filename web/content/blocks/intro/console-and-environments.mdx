## bin/console

`bin/console` — это CLI-приложение вашего проекта, построенное на компоненте `symfony/console`.
Через него запускаются как встроенные команды фреймворка, так и ваши собственные.

```bash
php bin/console list                # список всех доступных команд
php bin/console about               # версия Symfony, окружение, PHP и т.д.
php bin/console debug:router        # таблица всех зарегистрированных роутов
php bin/console debug:container     # список сервисов в DI-контейнере
php bin/console cache:clear         # очистить скомпилированный кэш
php bin/console make:controller     # (если поставлен MakerBundle) сгенерировать контроллер
```

Каждая своя команда — это обычный PHP-класс с атрибутом `#[AsCommand]`, который наследуется от
`Command` и переопределяет метод `execute()`. Мы напишем такую команду, когда дойдём до отдельного
блока про Console.

## Окружения: dev, prod, test

Symfony-приложение может работать в нескольких **окружениях** (environments), которые управляются
переменной `APP_ENV`:

- **dev** — окружение для разработки. Включён профайлер (web debug toolbar), подробные страницы
  ошибок со stack trace, конфигурация автоматически пересобирается при каждом изменении файлов.
- **prod** — production-окружение. Никакой отладочной информации не показывается пользователю,
  весь кэш (роуты, контейнер сервисов, конфигурация) заранее скомпилирован для максимальной
  скорости — это одна из причин, почему Symfony остаётся быстрым несмотря на свою «тяжеловесность».
- **test** — используется при прогоне автотестов (PHPUnit + `KernelTestCase`/`WebTestCase`),
  обычно с отдельной тестовой базой данных.

Одно и то же приложение с одним и тем же кодом ведёт себя по-разному в зависимости от `APP_ENV` —
это не отдельные копии кода, а разный набор включённых бандлов и разная стратегия кэширования.

![Три окружения: dev — профайлер и подробные ошибки, prod — всё закэшировано, test — для автотестов](/images/intro/three-environments.png)

## .env файлы

Переменные окружения Symfony читает из каскада файлов, в порядке возрастания приоритета:

1. `.env` — значения по умолчанию, **коммитится** в git;
2. `.env.local` — локальные переопределения конкретного разработчика, **в git не попадает**;
3. `.env.$APP_ENV` (например, `.env.test`) — специфичные для окружения значения, коммитятся;
4. `.env.$APP_ENV.local` — локальные переопределения для конкретного окружения, в git не попадает.

Типичный пример: в `.env` лежит `DATABASE_URL` с параметрами локальной SQLite/MySQL по умолчанию,
а у каждого разработчика в `.env.local` — свои реальные креды от базы.

![.env — каскад приоритетов: .env, .env.local, .env.$APP_ENV, .env.$APP_ENV.local](/images/intro/env-cascade.png)

## Кэш и контейнер — коротко

При первом запросе (или явном `cache:warmup`) Symfony компилирует всю конфигурацию сервисов в
один большой сгенерированный PHP-класс — скомпилированный DI-контейнер. Это значит, что в
production не происходит никакого «магического» поиска сервисов на каждый запрос — просто
вызывается метод сгенерированного класса. Подробно контейнер зависимостей разберём в отдельном
блоке, здесь достаточно знать, что он **существует** и что `bin/console cache:clear` — безопасный
способ заставить его пересобраться с нуля.
