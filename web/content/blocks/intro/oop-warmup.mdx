Прежде чем переходить к роутингу и контроллерам, стоит закрепить ту часть ООП в PHP, на которой
буквально всё держится в Symfony — потому что «магии» там на самом деле нет.

## Классы — это и есть сервисы

Когда вы слышите «сервис» в контексте Symfony, за этим не стоит ничего особенного: сервис — это обычный
PHP-объект, который умеет что-то делать, и который Symfony создаёт за вас и передаёт туда, где он
нужен (это называется **Dependency Injection**, разберём подробно в отдельном блоке).

```php
namespace App\Service;

class BookService
{
    public function formatTitle(string $title): string
    {
        return mb_strtoupper($title) . '!';
    }
}
```

Позже в контроллере вы просто укажете этот класс как аргумент конструктора или метода — и Symfony
сам создаст объект `BookService` и передаст его вам. Никакой ручной сборки `new BookService()` в
контроллерах писать не придётся.

## Namespace и автозагрузка через Composer

Файл `App\Service\BookService` физически лежит в `src/Service/BookService.php`. Это не совпадение:
Symfony-проект настроен на **PSR-4 автозагрузку** через `composer.json`:

```json
{
  "autoload": {
    "psr-4": {
      "App\\": "src/"
    }
  }
}
```

Это значит: неймспейс `App\` соответствует каталогу `src/`, а дальше — один в один. Класс
`App\Service\BookService` обязан лежать в `src/Service/BookService.php`. Благодаря этому правилу
Composer генерирует автозагрузчик, и вам никогда не приходится писать `require` вручную.

## Практика

Допиши метод `formatTitle()` в классе ниже так, чтобы он возвращал заголовок книги в верхнем
регистре с восклицательным знаком в конце. Обрати внимание на кириллицу — обычный `strtoupper()`
с ней не работает.
