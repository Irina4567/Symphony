Последний штрих перед мини-проектом: что показать пользователю сразу после успешного действия —
и почему это стоит делать через редирект, а не просто вывести результат на той же странице.

## Post/Redirect/Get

```php
if ($form->isSubmitted() && $form->isValid()) {
    $em->persist($book);
    $em->flush();

    return $this->redirectToRoute('books_index'); // редирект, не render()
}
```

Если вместо редиректа отрендерить страницу с результатом прямо в ответ на `POST`, у пользователя
в браузере в истории останется именно этот POST-запрос. Нажмёт `F5` — браузер честно
переспросит «отправить данные ещё раз?», и невнимательный клик может создать вторую книгу.
Редирект на `GET`-страницу решает эту проблему: после него в адресной строке — безопасный `GET`,
который можно обновлять сколько угодно.

## Flash-сообщения: передать информацию через редирект

Проблема: после `redirectToRoute()` мы теряем контекст — новая страница ничего не знает про то,
что только что произошло успешное создание книги. Для этого и существуют **flash-сообщения** —
однократные сообщения, которые кладут в сессию перед редиректом, а следующая страница их читает
и они тут же удаляются.

```php
$this->addFlash('success', 'Книга добавлена!');

return $this->redirectToRoute('books_index');
```

```twig
{% for message in app.flashes('success') %}
    <div class="alert alert-success">{{ message }}</div>
{% endfor %}
```

Категория (`'success'`, `'error'`, `'warning'`) — обычное соглашение для стилизации в шаблоне,
самому фреймворку эта строка ничего не диктует.

Ту же самую сумку с сообщениями можно прочитать и без Twig, прямо в контроллере — именно так
устроен маршрут `/exercises/flash-show`, на который ссылается упражнение ниже:

```php
$messages = $request->getSession()->getFlashBag()->get('success');
```

`app.flashes('success')` в Twig — просто удобная обёртка над тем же самым вызовом.

## Почему это работает только с сессией

Flash-сообщение живёт в сессии — а сессия существует только пока клиент присылает тот же самый
cookie (`PHPSESSID` по умолчанию), который сервер выдал в предыдущем ответе. В браузере это
происходит само собой; в упражнениях этого урока — благодаря общему cookie jar, который
песочница держит на всю последовательность запросов одного упражнения, как если бы это был один
и тот же браузер, переходящий со страницы на страницу.
