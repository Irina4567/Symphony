# Модуль 33. Миграция legacy-проектов и апгрейд мажорных версий

> Предыдущий модуль: [32 — Многотенантность и масштабирование БД](32-multitenancy-i-masshtabirovanie.md)
>
> Большая часть работы senior-разработчика в реальности — не написание нового кода с чистого листа, а эволюция существующих систем. Этот навык почти никогда не преподаётся на учебных проектах, но именно он определяет зарплатную вилку senior/lead.

## 33.1. Апгрейд минорных версий Symfony (например, 7.1 → 7.2)

Symfony следует строгому semver: минорные версии (в рамках одной мажорной) **не содержат breaking changes** — только новый функционал и **deprecations** (пометки "этот способ устареет в будущем, используйте новый уже сейчас").

### Процесс апгрейда минорной версии

```bash
composer require symfony/symfony:^7.2
```

```bash
# ключевой инструмент — прогнать тесты с включённым отображением deprecation-предупреждений
SYMFONY_DEPRECATIONS_HELPER=weak php bin/phpunit
```

`SYMFONY_DEPRECATIONS_HELPER=weak` не проваливает тесты из-за deprecation-предупреждений, но **показывает** их количество — так вы видите, сколько мест кода использует устаревающий API, не блокируя CI прямо сейчас, но получая видимость технического долга.

```bash
php bin/console debug:container --deprecations   # аналогично из модуля 20
```

**Практика "апгрейд на каждый минорный релиз, не откладывая"** — критически важна: если игнорировать deprecations в течение нескольких лет и накопить их сотнями, апгрейд превращается в многомесячный проект вместо рутинной еженедельной задачи.

## 33.2. Апгрейд мажорной версии (например, Symfony 6 → 7)

Мажорные версии **удаляют** то, что было помечено deprecated в последней минорной версии предыдущей мажорной ветки. Поэтому официальная рекомендуемая стратегия:

1. **Обновиться до последней минорной версии текущей мажорной** (например, 6.4 — последняя минорная перед 7.0) — LTS-релизы специально существуют как "стабильная точка" для этого шага.
2. **Устранить все deprecation-предупреждения** на этой версии (они точно указывают, что изменится в следующей мажорной).
3. **Только после этого** переключаться на новую мажорную версию — к этому моменту в идеале не должно остаться ни одной причины, по которой апгрейд что-то сломает.

```bash
composer require symfony/symfony:^6.4  # финальный шаг перед переходом на 7.x
# устранить ВСЕ deprecations
composer require symfony/symfony:^7.0  # теперь апгрейд должен пройти практически без сюрпризов
```

### Rector для автоматизации мажорного апгрейда

Мы уже касались Rector в модуле 18 — именно для этого сценария он особенно ценен:

```php
// rector.php
return static function (RectorConfig $rectorConfig): void {
    $rectorConfig->sets([
        SymfonySetList::SYMFONY_64,
        SymfonySetList::SYMFONY_70,
        SymfonySetList::SYMFONY_CODE_QUALITY,
    ]);
};
```

```bash
vendor/bin/rector process --dry-run  # обзор изменений
vendor/bin/rector process             # применение
```

Rector автоматизирует значительную часть механических изменений (переименованные методы, изменённые сигнатуры), оставляя вам фокус на действительно требующих человеческого решения местах.

## 33.3. UPGRADE.md — не пропускайте официальный чеклист

Каждая мажорная (и минорная) версия Symfony сопровождается файлом `UPGRADE-X.Y.md` в официальном репозитории — исчерпывающий список **всех** breaking changes с конкретными инструкциями "было → стало". Это первое, что должен прочитать senior-разработчик перед апгрейдом, а не полагаться только на автоматизацию (Rector покрывает многое, но не всё, особенно поведенческие изменения, а не просто синтаксические).

## 33.4. Стратегия "Strangler Fig" для миграции целого legacy-приложения

Когда речь не об апгрейде версии Symfony, а о **миграции** с совершенно другой технологии (старый самописный PHP без фреймворка, устаревший Zend Framework, монолит на другом стеке) — переписывание "с нуля одним большим релизом" почти всегда заканчивается провалом (месяцы/годы разработки без видимого бизнес-результата, расхождение нового и старого кода, "второй системный эффект").

**Strangler Fig Pattern** (назван по аналогии с растением-душителем, постепенно обвивающим дерево-опору) — постепенная замена частей legacy-системы новыми, работающими **параллельно** со старым кодом, через общий "фасад"-роутер:

```
                     ┌──────────────┐
Пользователь ──────▶ │ Reverse Proxy │
                     │  (Nginx/Traefik) │
                     └──────────────┘
                        │            │
              /catalog/*│            │остальное
                        ▼            ▼
                 ┌─────────────┐  ┌──────────────┐
                 │ Новый Symfony │  │  Старое legacy │
                 │  (постепенно  │  │   приложение   │
                 │   растёт)      │  │  (постепенно   │
                 │                 │  │   сжимается)   │
                 └─────────────┘  └──────────────┘
```

```nginx
# постепенная маршрутизация — каждый переписанный раздел переключается отдельно
location /catalog/ {
    proxy_pass http://symfony-app;
}
location /checkout/ {
    proxy_pass http://symfony-app;
}
location / {
    proxy_pass http://legacy-app; # всё остальное пока идёт в старую систему
}
```

Ключевое преимущество: каждый переписанный раздел **сразу приносит пользу**, риск изолирован (если новый `/catalog` содержит баг, откатывается только переключение этого раздела в конфиге прокси, а не весь проект), команда получает обратную связь от реальных пользователей на каждом этапе, а не через год "большого взрыва".

### Общая сессия/аутентификация между старой и новой системой (частая сложность)

Обычно решается через общее хранилище сессий (Redis, к которому оба приложения имеют доступ) с одинаковым форматом токена/куки, либо через полноценный SSO (модуль 26), выделяемый как самый первый мигрируемый компонент — тогда обе системы сразу становятся независимы друг от друга в вопросе аутентификации.

## 33.5. Работа с legacy-кодом внутри Symfony-проекта

Даже сам Symfony-проект со временем накапливает legacy-участки (старый код, который "работает, но страшно трогать"). Практики безопасной эволюции:

### Characterization Tests — тесты, документирующие текущее (не обязательно правильное) поведение

Прежде чем рефакторить непонятный legacy-метод без тестов, сначала напишите тест, **фиксирующий его текущее поведение** (даже если оно кажется странным) — это даёт защитную сеть перед последующим рефакторингом:

```php
public function testLegacyDiscountCalculationCurrentBehavior(): void
{
    // Мы не уверены, что эта логика "правильная", но фиксируем ЧТО ЕСТЬ,
    // прежде чем начинать её трогать — так рефакторинг не сломает существующих клиентов
    $result = (new LegacyDiscountCalculator())->calculate(1000, 'VIP');
    self::assertSame(850, $result); // документируем текущий результат, даже если логика неочевидна
}
```

### Strangler внутри одного модуля — постепенный редирект вызовов

```php
class DiscountCalculator
{
    public function __construct(
        private LegacyDiscountCalculator $legacy,
        private NewDiscountPolicy $new,
        #[Autowire('%feature.new_discount_policy_enabled%')] private bool $useNewPolicy,
    ) {}

    public function calculate(int $price, string $customerTier): int
    {
        if ($this->useNewPolicy) {
            return $this->new->apply($price, $customerTier);
        }
        return $this->legacy->calculate($price, $customerTier); // старый путь остаётся доступным для отката
    }
}
```

Feature flag (`feature.new_discount_policy_enabled`) позволяет включить новую реализацию для части трафика (canary), сравнить результаты старой/новой логики в production на реальных данных перед полным переключением, и мгновенно откатиться при обнаружении расхождений — без нового деплоя.

## 33.6. Практика: гипотетическая миграция BookNest с legacy-платформы

Представим, что BookNest изначально существовал как обычный PHP-скрипт без фреймворка ("php-спагетти"), и мы переводим его на Symfony поэтапно:

1. **Этап 0** — выделить общую аутентификацию через Redis-сессии, доступные обеим системам.
2. **Этап 1** — переписать каталог (`/catalog/*`) на Symfony, остальное остаётся на legacy, маршрутизация через Nginx.
3. **Этап 2** — переписать оформление заказа, используя уже существующую в Symfony модель `Book`/`Author` из этапа 1.
4. **Этап 3** — переписать админ-панель.
5. **Этап 4** — legacy-приложение полностью выведено из эксплуатации, Nginx-конфиг маршрутизации удалён (весь трафик идёт на Symfony).

## 33.8. Частые ошибки

1. **Откладывают апгрейд минорных версий на годы** — деprecated-API накапливается в геометрической прогрессии, финальный апгрейд превращается в отдельный многомесячный проект вместо рутины.
2. **Пытаются переписать legacy-систему "одним большим релизом"** — классическая ловушка, статистически чаще проваливается, чем удаётся.
3. **Рефакторят legacy-код без тестов "на глаз"**, полагаясь на код-ревью вместо автоматической защитной сети — тихие поведенческие регрессии, не замеченные до продакшена.
4. **Не читают `UPGRADE.md`**, полагаясь только на автоматизированные инструменты (Rector) — пропускают поведенческие изменения, которые невозможно автоматически детектировать синтаксическим анализом.
5. **Мигрируют "по техническому удобству"**, а не по бизнес-ценности — переписывают простые, редко используемые разделы вместо критичных для бизнеса частей, откладывая реальную ценность миграции.
