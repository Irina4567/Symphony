## Часть 1. Многотенантность (Multi-tenancy)

## 32.1. Что такое multi-tenancy и когда она нужна

**Multi-tenancy** — архитектура, при которой одно приложение обслуживает **множество изолированных клиентов** (tenants) — например, если бы BookNest продавался как SaaS-платформа "создай свой книжный интернет-магазин", и каждый владелец магазина видел бы только свои книги, заказы, покупателей.

### Три классические модели изоляции

**1. Отдельная база данных на каждого тенанта** — максимальная изоляция (проще всего гарантировать, что данные одного клиента физически не могут "утечь" к другому), но дороже в эксплуатации (миграции нужно прогонять на N баз, сложнее шарить общие справочники).

```php
class TenantDatabaseResolver
{
    public function resolveConnectionParams(string $tenantId): array
    {
        return ['dbname' => "booknest_tenant_$tenantId", /* ... */];
    }
}
```

**2. Общая БД, отдельная схема (schema) на тенанта** (актуально для PostgreSQL) — компромисс: изоляция на уровне БД без overhead полностью раздельных инстансов.

**3. Общая БД и таблицы, разделение по `tenant_id`** — самый дешёвый в эксплуатации вариант (одна миграция на всех, один connection pool), но требует **дисциплины на уровне кода**: каждый запрос обязан фильтроваться по `tenant_id`, иначе — утечка данных между клиентами.

## 32.2. Реализация "общая БД + tenant_id" через Doctrine Filter

Самый частый в реальной практике подход. Ключевая опасность — забыть добавить `WHERE tenant_id = ?` хотя бы в одном запросе. Doctrine предоставляет механизм **SQL Filters**, применяющийся автоматически ко **всем** запросам:

```php
<?php

namespace App\Doctrine;

use Doctrine\ORM\Mapping\ClassMetadata;
use Doctrine\ORM\Query\Filter\SQLFilter;

class TenantFilter extends SQLFilter
{
    public function addFilterConstraint(ClassMetadata $targetEntity, string $targetTableAlias): string
    {
        if (!$targetEntity->reflClass->implementsInterface(TenantAwareInterface::class)) {
            return '';
        }

        $tenantId = $this->getParameter('tenant_id'); // уже безопасно экранировано Doctrine

        return sprintf('%s.tenant_id = %s', $targetTableAlias, $tenantId);
    }
}
```

```yaml
# config/packages/doctrine.yaml
doctrine:
    orm:
        filters:
            tenant_filter:
                class: App\Doctrine\TenantFilter
                enabled: false   # включаем явно за запрос, ниже
```

```php
#[AsEventListener(event: KernelEvents::REQUEST, priority: 10)]
class EnableTenantFilterListener
{
    public function __construct(private EntityManagerInterface $em, private TenantResolver $tenantResolver) {}

    public function __invoke(RequestEvent $event): void
    {
        $tenantId = $this->tenantResolver->resolve($event->getRequest());

        $filter = $this->em->getFilters()->enable('tenant_filter');
        $filter->setParameter('tenant_id', $tenantId);
    }
}
```

После этого **любой** Doctrine-запрос (включая написанные через QueryBuilder в любом репозитории) автоматически получает `AND tenant_id = ?` — разработчик физически не может "забыть" фильтр, потому что он применяется на уровне ORM, а не на уровне каждого отдельного запроса.

**Критично**: не забудьте отключать/переключать `tenant_id` при выполнении фоновых задач (Messenger-обработчики, консольные команды) — там нет HTTP-запроса, из которого можно определить текущего тенанта, поэтому tenant должен передаваться явно в сообщении/аргументе команды.

```php
#[AsMessageHandler]
class SendOrderConfirmationHandler
{
    public function __invoke(SendOrderConfirmationMessage $message): void
    {
        $this->tenantContext->setCurrentTenant($message->tenantId); // явная установка вне HTTP-контекста
        $filter = $this->em->getFilters()->enable('tenant_filter');
        $filter->setParameter('tenant_id', $message->tenantId);
        // ...
    }
}
```

## 32.3. Определение текущего тенанта

```php
class TenantResolver
{
    public function resolve(Request $request): string
    {
        // по поддомену: shop1.booknest.example → tenant "shop1"
        $host = $request->getHost();
        $subdomain = explode('.', $host)[0];

        return $subdomain;

        // альтернативы: заголовок X-Tenant-Id (для API), путь /tenants/{id}/..., атрибут авторизованного пользователя
    }
}
```

## Часть 2. Масштабирование базы данных

## 32.4. Read Replicas — разделение чтения и записи

Когда один инстанс БД перестаёт справляться с нагрузкой чтения (что для типичного веб-приложения — основная доля запросов, записи заметно меньше), классическое решение — **реплики только для чтения** (read replicas), с асинхронной репликацией с основного (primary/master) сервера.

```yaml
# config/packages/doctrine.yaml
doctrine:
    dbal:
        connections:
            default:
                url: '%env(DATABASE_URL)%'         # primary — для записи
                replicas:
                    replica1:
                        url: '%env(DATABASE_REPLICA_URL)%'   # только для чтения
```

Doctrine автоматически направляет `SELECT`-запросы на реплику, а `INSERT`/`UPDATE`/`DELETE` — на primary. **Важный нюанс** — **репликационный лаг** (replication lag): реплика обновляется с небольшой задержкой (обычно миллисекунды, но может быть больше под нагрузкой). Это означает возможность ситуации "только что создал заказ → сразу читаю список заказов → нового заказа ещё нет в списке", потому что чтение попало на ещё не догнавшую реплику.

### Решения проблемы репликационного лага

```php
// принудительное чтение с primary сразу после записи в рамках одного запроса
$connection->executeStatement('SET SESSION default_transaction_read_only = off');
// или явно указать Doctrine использовать primary-соединение для конкретного запроса
```

Практический паттерн: сразу после создания записи, если её нужно немедленно прочитать (например, вернуть в JSON-ответе API), **не читайте её заново из БД** — верните тот же объект, который только что был создан в памяти (Doctrine и так уже "знает" все его поля, включая сгенерированный `id`).

## 32.5. Connection Pooling

Каждое новое соединение с БД имеет накладные расходы (TCP handshake, аутентификация). При высокой конкурентности множество PHP-FPM/worker процессов, открывающих собственные соединения, могут исчерпать лимит подключений БД (`max_connections` в PostgreSQL/MySQL).

**PgBouncer** (для PostgreSQL) / **ProxySQL** (для MySQL) — прокси-серверы, поддерживающие пул уже открытых соединений к реальной БД, переиспользуемых между короткоживущими соединениями от приложения:

```
PHP-FPM (100 воркеров, каждый мог бы открыть своё соединение)
       │
       ▼
   PgBouncer (пул из, скажем, 20 реальных соединений)
       │
       ▼
   PostgreSQL (спокойно работает с 20 постоянными соединениями вместо 100 нестабильных)
```

```dotenv
DATABASE_URL="postgresql://app:pass@pgbouncer:6432/app?serverVersion=16&charset=utf8"
```

Особенно критично в связке с worker mode раннерами (модуль 29) и serverless-окружениями, где количество параллельных PHP-процессов может быть большим и непредсказуемым.

## 32.6. Sharding — горизонтальное разделение данных

Когда даже с репликами и pooling один физический сервер БД перестаёт справляться с объёмом данных/записи (это происходит на масштабах, значительно превышающих типичный проект — обычно сотни миллионов+ строк или экстремальная нагрузка записи) — данные разделяют по нескольким независимым серверам БД (**шардам**), каждый из которых хранит **часть** данных.

```php
class ShardResolver
{
    public function resolveShardForCustomer(int $customerId): string
    {
        // простейшая стратегия — по остатку от деления
        $shardIndex = $customerId % 4; // 4 шарда
        return "shard_$shardIndex";
    }
}
```

**Это самое дорогое и сложное архитектурное решение** из всех рассмотренных в модуле — оно кардинально усложняет любые кросс-shard операции (JOIN между данными разных клиентов, агрегированная аналитика по всей системе, транзакции, затрагивающие несколько шардов). Symfony/Doctrine не имеет "готовой из коробки" поддержки sharding — требуется существенная кастомная инфраструктура (собственный `ShardManager`, логика роутинга запросов, стратегия ресардинга при росте). **Прежде чем внедрять sharding, всегда рассмотрите более простые альтернативы**: вертикальное масштабирование (более мощный сервер), оптимизация запросов и индексов (модуль 6), кэширование (модуль 15), read replicas — sharding нужен реальному "хайлоуду", а не среднему проекту "на будущее".

## 32.7. Практика: подготовка BookNest к SaaS-модели

Гипотетический сценарий: превращаем BookNest в платформу "создай свой книжный магазин" для разных владельцев (tenants), используя модель "общая БД + `tenant_id`":

1. Добавить `tenant_id` во все "клиентские" таблицы (`books`, `orders`, `customers` — но НЕ в общие справочники вроде глобального каталога жанров, если он общий для всех).
2. Реализовать `TenantResolver` по поддомену.
3. Настроить `TenantFilter`, автоматически применяемый ко всем запросам.
4. Убедиться, что фоновые Messenger-обработчики (модуль 14) явно передают `tenant_id` в сообщении.
5. Написать тест, специально проверяющий **невозможность** утечки данных между тенантами (создать данные для tenant A, попытаться прочитать их под контекстом tenant B, убедиться в пустом результате).

## 32.9. Частые ошибки

1. **Забывают `tenant_id` фильтр хотя бы в одном запросе** при ручной реализации (без Doctrine Filter) — классическая и крайне опасная уязвимость утечки данных между клиентами.
2. **Не передают tenant-контекст в асинхронные обработчики** — фоновые задачи падают или (хуже) работают с неверным тенантом.
3. **Читают только что записанные данные с read-реплики** без учёта репликационного лага — пользователь не видит только что созданный им заказ.
4. **Внедряют sharding преждевременно** — усложняют архитектуру на порядок раньше реальной необходимости.
5. **Не тестируют изоляцию тенантов явно** — полагаются на "должно работать", хотя это одна из самых критичных вещей для проверки именно тестами, а не ревью кода на глаз.
