## 29.1. Модель "shared-nothing" классического PHP-FPM

Традиционная модель выполнения PHP: каждый HTTP-запрос обрабатывается **с чистого листа** — процесс PHP-FPM загружает весь код (пусть и из OPcache-байткода), создаёт все объекты заново, а в конце запроса **вся память полностью освобождается**. Это называется "shared-nothing" архитектурой.

**Преимущество**: исключительная надёжность и простота ментальной модели — утечка памяти в одном запросе не влияет на следующий, поскольку процесс/память "с нуля" на каждый запрос (в рамках воркера, который переиспользуется, но состояние приложения не сохраняется между запросами).

**Недостаток**: накладные расходы на повторную инициализацию (bootstrapping) фреймворка при каждом запросе — даже с максимально прогретым OPcache, Symfony Kernel всё равно создаётся заново, конфигурация читается заново (хоть и из скомпилированного кэша).

## 29.2. OPcache — фундаментальная оптимизация, о которой уже говорили

Мы разбирали базовую настройку OPcache в модуле 17. Углубим понимание.

### JIT (Just-In-Time компиляция) — PHP 8+

```ini
opcache.jit=1255
opcache.jit_buffer_size=128M
```

JIT компилирует "горячий" код в машинный код на лету. **Важная практическая деталь**: JIT даёт заметный прирост на **CPU-bound** нагрузке (сложные математические вычисления, обработка изображений, парсинг) — но почти не влияет на типичное Symfony-приложение, где основное время уходит на I/O (SQL-запросы, HTTP-вызовы, чтение файлов), а не вычисления. Не ждите магического ускорения от одного включения JIT для обычного CRUD-приложения — измеряйте на своей нагрузке.

### Preloading — предзагрузка классов при старте PHP-FPM

```ini
opcache.preload=/var/www/booknest/config/preload.php
opcache.preload_user=www-data
```

```php
<?php
// config/preload.php
require dirname(__DIR__).'/vendor/autoload.php';

$preloaded = 0;
foreach ((new \Symfony\Component\Finder\Finder())->in(__DIR__.'/../src')->name('*.php')->files() as $file) {
    try {
        opcache_compile_file($file->getPathname());
        $preloaded++;
    } catch (\Throwable) {
        // некоторые файлы (например, с syntax зависимой от окружения) могут не преload-иться — это нормально
    }
}
```

Preloading загружает и компилирует классы в память **один раз при старте PHP-FPM master-процесса**, а не при первом запросе к каждому воркеру. Даёт заметный выигрыш на приложениях с большим количеством классов (что типично для Symfony с его тысячами сервисов/классов от Doctrine, Validator и т.д.).

**Важный нюанс**: preloaded-классы остаются в памяти **до перезапуска PHP-FPM** — обновление кода на диске не подхватится без полного рестарта процесса (`systemctl restart php8.3-fpm`), в отличие от обычного OPcache с `validate_timestamps`.

## 29.3. Альтернативные раннеры: зачем и когда

Альтернатива классической shared-nothing модели — **worker mode**: PHP-процесс запускается один раз и обрабатывает **множество** запросов подряд, не завершаясь между ними (похоже на модель Node.js/Java). Кандидаты: **Swoole**, **RoadRunner**, **FrankenPHP** (написан на Go, включает Caddy-сервер, официально поддерживается Symfony с 2023 года как "родной" вариант).

```bash
# FrankenPHP — самый простой путь для Symfony сейчас
composer require frankenphp
```

```bash
frankenphp php-server --worker public/index.php
```

### Преимущества worker mode

- Kernel, DI-контейнер и другие "тяжёлые" объекты создаются **один раз**, а не на каждый запрос — существенно ниже latency при высокой нагрузке.
- Возможность держать долгоживущие соединения (WebSocket) — то, что мы обсуждали в модуле 25 как альтернативу Mercure для сложных двусторонних сценариев.
- Меньше накладных расходов на процесс в целом (нет постоянного форка/уничтожения процессов PHP-FPM).

### Цена — statefulness

Это тот самый нюанс, о котором мы предупреждали ещё в модуле 4 (раздел 4.3): **singleton-сервисы теперь живут между запросами** в рамках одного воркера. Всё, что раньше "автоматически" сбрасывалось между запросами (потому что процесс уничтожался целиком), теперь может **утечь** из одного запроса в следующий, если явно не сброшено.

```php
// ОПАСНО в worker mode: статическое свойство "запоминает" значение между запросами
class RequestScopedCache
{
    private array $cache = []; // singleton-сервис — этот массив НЕ очищается автоматически между запросами!

    public function remember(string $key, callable $factory): mixed
    {
        return $this->cache[$key] ??= $factory();
    }
}
```

### Как правильно готовить сервисы к worker mode

```yaml
# сервисы, требующие сброса состояния между запросами, помечаются как "resettable"
services:
    App\Service\RequestScopedCache:
        tags: ['kernel.reset', 'reset_on_terminate']
```

```php
use Symfony\Contracts\Service\ResetInterface;

class RequestScopedCache implements ResetInterface
{
    private array $cache = [];

    public function reset(): void
    {
        $this->cache = []; // вызывается автоматически после каждого запроса в worker mode
    }
}
```

Symfony имеет встроенный механизм `kernel.reset` — сервисы, реализующие `ResetInterface` и помеченные соответствующим тегом, автоматически сбрасываются между запросами при работе в worker mode (это же используется во внутренних сервисах фреймворка — Doctrine `EntityManager` тоже "сбрасывается" через похожий механизм, потому что иначе Unit of Work бы бесконтрольно рос память от запроса к запросу).

## 29.4. Практический чеклист готовности приложения к worker mode

1. **Аудит singleton-сервисов** на предмет накопления состояния (массивы, кэши в памяти объекта) — либо реализовать `ResetInterface`, либо сделать сервис `shared: false`.
2. **EntityManager** требует явного `$em->clear()` между запросами в некоторых конфигурациях worker mode (в зависимости от интеграции — уточняйте документацию конкретного раннера, так как поддержка развивается).
3. **Утечки памяти в сторонних библиотеках** — воркер, работающий сутками, накапливает даже небольшие утечки. Настраивайте `max_requests`/`time-limit` для периодического перезапуска воркеров (аналог того, что мы делали для Messenger-воркеров в модуле 14).
4. **Глобальное состояние** (`static` свойства, глобальные переменные) — требует особого внимания, это самый частый источник багов "работает в первом запросе, ломается во втором".
5. **Файловые дескрипторы и соединения** (не закрытые вручную) — в shared-nothing модели ОС закрывала их при завершении процесса, в worker mode это уже не происходит автоматически.

## 29.5. Когда worker mode действительно нужен

**Не переходите на worker mode "потому что это быстрее" без измерений.** Для большинства Symfony-приложений с типичной нагрузкой (десятки-сотни запросов в секунду) классический PHP-FPM с хорошо настроенным OPcache и оптимизированными SQL-запросами (модули 5-6) справляется отлично, а операционная простота shared-nothing модели того стоит. Worker mode оправдан, когда:

- Профилирование (Blackfire/Xdebug) показывает, что заметная часть latency уходит именно на bootstrap фреймворка, а не на бизнес-логику/БД.
- Действительно высокая нагрузка (тысячи RPS), где экономия на bootstrap кумулятивно даёт существенный эффект.
- Нужны долгоживущие соединения (свой WebSocket-сервер, что мы обсуждали в модуле 25).

## 29.6. Дополнительные приёмы производительности

### Lazy Services — отложенная инициализация "тяжёлых" сервисов

```php
#[Autowire(lazy: true)]
private ReportGeneratorInterface $reportGenerator;
```

Если сервис используется только в редких сценариях (например, генерация годового отчёта), но конструктор внедряется всегда, `lazy: true` создаёт proxy-объект, который реально инстанцирует сервис только при первом вызове метода — экономит время инициализации для запросов, где сервис не используется.

```bash
composer require symfony/proxy-manager-bridge  # требуется для lazy-сервисов
```

### Профилирование в production без Xdebug

Xdebug слишком дорог по производительности для production. Для боевого профилирования используются легковесные инструменты:
- **Blackfire** — специально спроектирован для production-профилирования с минимальным оверхедом, "заточен" под PHP/Symfony.
- **XHProf/Tideways** — альтернативы с похожим назначением.

```php
// точечное профилирование конкретного узкого места без полноценного APM
$start = hrtime(true);
$result = $this->expensiveOperation();
$this->logger->info('expensiveOperation took ' . (hrtime(true) - $start) / 1e6 . 'ms');
```

### Composer autoloader — финальная проверка

```bash
composer dump-autoload --optimize --classmap-authoritative --no-dev
```

`--classmap-authoritative` — самый агрессивный режим: Composer считает class-map полной и **не** пытается искать классы на диске, если их нет в карте — небольшой, но измеримый выигрыш на каждый запрос, особенно заметный при большом количестве классов.

## 29.8. Частые ошибки

1. **Включают worker mode без аудита state-full сервисов** — production-баги "первый пользователь видит данные второго", крайне неприятные и трудноуловимые.
2. **Ожидают от JIT ускорения обычного CRUD-приложения** — JIT почти не помогает I/O-bound нагрузке, которой является большинство веб-приложений.
3. **Используют Xdebug в production "для отладки production-проблемы"** — катастрофическое падение производительности под реальной нагрузкой.
4. **Не настраивают периодический рестарт долгоживущих воркеров** — постепенная деградация из-за небольших утечек памяти в сторонних библиотеках, накапливающихся за дни работы.
5. **Меняют раннер (PHP-FPM → worker mode) без нагрузочного тестирования "до/после"** — принимают архитектурное решение вслепую, без данных.
