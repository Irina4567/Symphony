## 31.1. Mutation Testing — тестирование тестов

Классическая проблема: высокий процент покрытия кода (модуль 13) **не гарантирует**, что тесты реально проверяют поведение. Тест может формально "выполнить" строку кода, ничего толком не проверив (например, забыли assert).

**Infection** — инструмент mutation testing: он автоматически вносит небольшие искусственные "мутации" в ваш код (меняет `>` на `>=`, `true` на `false`, удаляет вызов метода) и запускает тесты. Если тесты **всё ещё проходят** после мутации — значит, эта часть кода **не была реально протестирована**, несмотря на формальное покрытие.

```bash
composer require --dev infection/infection
```

```json
// infection.json5
{
    "source": { "directories": ["src"] },
    "logs": { "text": "infection.log" },
    "mutators": { "@default": true }
}
```

```bash
vendor/bin/infection --threads=4 --min-msi=70
```

### Пример "выжившей" мутации

```php
// исходный код
public function isEligibleForFreeShipping(int $totalKopecks): bool
{
    return $totalKopecks >= 300000; // от 3000 рублей
}
```

```php
// тест
public function testFreeShipping(): void
{
    $service = new ShippingService();
    self::assertTrue($service->isEligibleForFreeShipping(400000));
}
```

Infection замутирует `>=` на `>` — и тест **всё ещё пройдёт** (400000 > 300000 тоже true)! Это значит, тест не проверяет граничное значение `300000` — мутация "выжила" (survived), сигнализируя дыру в тестовом покрытии. Исправление — добавить граничный тест:

```php
public function testExactThresholdIsEligible(): void
{
    self::assertTrue($service->isEligibleForFreeShipping(300000));
}

public function testJustBelowThresholdIsNotEligible(): void
{
    self::assertFalse($service->isEligibleForFreeShipping(299999));
}
```

**MSI (Mutation Score Indicator)** — процент "убитых" (обнаруженных тестами) мутаций. В отличие от простого покрытия кода, MSI — гораздо более честная метрика **качества** тестов, а не просто их наличия. Для критичной бизнес-логики (расчёт цены, скидок, налогов) разумная цель — MSI 80%+.

## 31.2. Нагрузочное тестирование

Функциональные тесты (модуль 13) проверяют **корректность**, но не **производительность под нагрузкой**. Приложение может быть абсолютно правильным логически и при этом "падать" при 100 одновременных пользователях из-за N+1-запросов, блокировок БД, недостаточного пула соединений.

### k6 — современный инструмент нагрузочного тестирования

```bash
brew install k6  # или docker run
```

```javascript
// load-test/checkout.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
    stages: [
        { duration: '30s', target: 50 },   // разогрев до 50 виртуальных пользователей
        { duration: '2m', target: 50 },    // держим нагрузку 2 минуты
        { duration: '30s', target: 200 },  // резкий всплеск до 200
        { duration: '1m', target: 0 },     // спад
    ],
    thresholds: {
        http_req_duration: ['p(95)<500'],  // 95% запросов должны укладываться в 500мс
        http_req_failed: ['rate<0.01'],     // не более 1% ошибок
    },
};

export default function () {
    const catalogResponse = http.get('https://booknest.example/catalog');
    check(catalogResponse, { 'catalog status is 200': (r) => r.status === 200 });

    sleep(1);

    const checkoutResponse = http.post('https://booknest.example/api/orders', JSON.stringify({
        items: [{ bookId: 1, quantity: 1 }],
    }), { headers: { 'Content-Type': 'application/json' } });

    check(checkoutResponse, { 'checkout status is 201': (r) => r.status === 201 });
}
```

```bash
k6 run load-test/checkout.js
```

### Что искать в результатах

- **p95/p99 latency** — не среднее (average скрывает "хвост" медленных запросов, которые реально раздражают пользователей), а перцентили.
- **Error rate под нагрузкой** — растёт ли % ошибок с ростом параллельных пользователей (признак исчерпания пула соединений БД, лимита файловых дескрипторов и т.д.).
- **Точка деградации** — при каком числе одновременных пользователей latency начинает расти нелинейно (это ваш реальный "потолок" текущей конфигурации, важно знать заранее, а не узнавать во время Black Friday-распродажи).

**Важно:** никогда не гоняйте нагрузочные тесты на production без предупреждения и координации — это, по сути, DDoS-атака на собственную систему. Используйте staging-окружение, максимально похожее на прод по конфигурации.

## 31.3. Contract Testing — для микросервисной архитектуры

Проблема: если BookNest API используется отдельной командой фронтенда/мобильной разработки (модуль 12), как убедиться, что изменение backend не сломает клиентов, **не дожидаясь** ручного интеграционного тестирования всей системы разом?

**Contract Testing** (например, через Pact) — потребитель API (consumer) описывает **ожидаемый контракт** ("я жду от `GET /api/books/1` такую структуру ответа"), контракт публикуется в общий Pact Broker, а поставщик API (provider, то есть BookNest) в своём CI **проверяет**, что реально отдаёт данные, соответствующие всем контрактам, заявленным потребителями — без необходимости деплоить и запускать оба сервиса вместе.


Это продвинутая техника, оправданная в первую очередь при **множестве независимых команд**, потребляющих один API — для монолитного приложения с единой командой обычно достаточно functional-тестов из модуля 13.

## 31.4. End-to-End тестирование через реальный браузер: Panther

Functional-тесты (`WebTestCase`, модуль 13) не выполняют JavaScript — они проверяют серверный рендеринг. Для проверки реального пользовательского сценария, включающего JS (Stimulus-контроллеры, Live Components, Turbo из модуля 16), нужен настоящий браузер.

```bash
composer require --dev symfony/panther
```

```php
<?php

namespace App\Tests\E2E;

use Symfony\Component\Panther\PantherTestCase;

class LiveSearchTest extends PantherTestCase
{
    public function testLiveSearchFiltersResultsWithoutPageReload(): void
    {
        $client = static::createPantherClient();
        $crawler = $client->request('GET', '/catalog');

        $client->waitFor('.book-card'); // дожидаемся, пока JS отрендерит начальный список

        $searchInput = $crawler->filter('input[data-model="query"]');
        $searchInput->sendKeys('Чистый код');

        $client->waitForVisibility('.book-card', 5); // Live Component успел ответить

        self::assertCount(1, $client->getCrawler()->filter('.book-card'));
        self::assertSelectorTextContains('.book-card', 'Чистый код');
    }
}
```

Panther запускает реальный headless-браузер (Chrome/Firefox через WebDriver) — самый медленный, но и самый достоверный уровень тестов (пирамида тестирования из модуля 13: используйте экономно, для критичных пользовательских сценариев целиком, не для каждой мелочи).

## 31.5. Тестирование производительности конкретного участка кода (benchmark)

```bash
composer require --dev phpbench/phpbench
```

```php
<?php

namespace App\Tests\Bench;

class PriceCalculatorBench
{
    private PriceCalculator $calculator;

    public function __construct()
    {
        $this->calculator = new PriceCalculator(taxRate: 0.20);
    }

    /**
     * @Revs(1000)
     * @Iterations(5)
     */
    public function benchCalculateWithTax(): void
    {
        $this->calculator->calculateWithTax(150000);
    }
}
```

```bash
vendor/bin/phpbench run --report=aggregate
```

Полезно при оптимизации "горячих" участков кода (часто вызываемых сервисов, кастомных алгоритмов) — позволяет объективно сравнить "было/стало" вместо субъективного "кажется, стало быстрее".

## 31.6. Практика: полный тестовый пирог BookNest

Сведём все уровни воедино для критичного модуля — `PriceCalculator` и `CheckoutService`:

1. **Unit-тесты** (модуль 13) — вся логика расчёта цены/скидок, изолированно.
2. **Mutation testing** — прогнать Infection на `src/Service/`, добиться MSI 80%+ для критичной бизнес-логики.
3. **Functional-тесты** (модуль 13) — полный сценарий оформления заказа через `WebTestCase`.
4. **E2E-тест** (Panther) — сценарий "поиск книги → добавление в корзину → оформление" через реальный браузер с JS.
5. **Нагрузочный тест** (k6) — `/checkout` под нагрузкой 100 одновременных пользователей, проверка p95 latency и отсутствия race condition из модуля 22 (двойное списание остатков товара).

## 31.8. Частые ошибки

1. **Гонятся за 100% покрытием кода**, не проверяя реальное качество тестов через mutation testing — высокое покрытие с низким MSI даёт ложное чувство защищённости.
2. **Запускают нагрузочные тесты прямо на production** без согласования — фактическая самоатака на систему.
3. **Пишут слишком много E2E-тестов** (Panther) вместо functional — тестовый набор становится мучительно медленным, CI занимает часы.
4. **Не проверяют p95/p99, а смотрят только на среднее время ответа** — упускают "хвост" медленных запросов, реально раздражающих часть пользователей.
5. **Внедряют Contract Testing для монолита с одной командой** — избыточная инфраструктурная сложность там, где достаточно обычных интеграционных тестов.
